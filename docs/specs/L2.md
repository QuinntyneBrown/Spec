# L2 Requirements — Spec CLI

---

## Traces to L1-1: CLI Application Foundation

### L2-1.1: Project Structure

The solution shall be a .NET 9 console application using the `System.CommandLine` library as the command-line parsing framework.

**Acceptance Criteria:**

- A .NET 9 console project exists with a reference to `System.CommandLine`.
- The application entry point configures and invokes a `RootCommand`.
- The application builds and runs without errors, displaying help text when invoked with `--help`.

### L2-1.2: Dependency Injection

The application shall use `Microsoft.Extensions.DependencyInjection` to compose services.

**Acceptance Criteria:**

- A `ServiceCollection` is configured and a `ServiceProvider` is built before command execution.
- Services required by command handlers are resolved from the DI container.

### L2-1.3: Logging

The application shall use `Microsoft.Extensions.Logging` for all diagnostic output.

**Acceptance Criteria:**

- An `ILogger` is registered in the DI container and injected into command handlers.
- Log output is written to the console at the configured log level.
- No command handler uses `Console.Write*` directly for diagnostic messages.

### L2-1.4: Configuration

The application shall use `Microsoft.Extensions.Configuration` to load settings.

**Acceptance Criteria:**

- Configuration is loaded from `appsettings.json` (if present) and environment variables.
- Configuration values are injectable via the DI container (e.g., `IOptions<T>` or `IConfiguration`).

## Traces to L1-2: Skill Installation Command

### L2-2.1: Install Command Registration

The CLI shall expose a command at `spec install` (or `install` subcommand) that triggers skill installation.

**Acceptance Criteria:**

- Running `spec install` executes the skill installation handler.
- Running `spec --help` lists `install` as an available command with a description.

### L2-2.2: Claude Code Skill File Generation

The `install` command shall create (or overwrite) a Claude Code skill file at `.claude/skills/spec.md` relative to the current working directory.

**Acceptance Criteria:**

- After running `spec install`, the file `.claude/skills/spec.md` exists.
- If `.claude/skills/` does not exist, it is created automatically.
- If the file already exists, it is overwritten with the latest version.
- The command logs a success message indicating the skill was installed.

### L2-2.3: Idempotent Installation

Running `spec install` multiple times shall produce the same result without errors.

**Acceptance Criteria:**

- Running `spec install` twice in succession completes successfully both times.
- The resulting `.claude/skills/spec.md` file is identical after both runs.

## Traces to L1-3: Spec Skill — Requirements Generation

### L2-3.1: Docs/Specs Folder Creation

The skill shall instruct the agent to create a `docs/specs` folder in the project root if it does not already exist.

**Acceptance Criteria:**

- The skill file contains an instruction to create `docs/specs` if missing.
- The instruction does not error or duplicate the folder if it already exists.

### L2-3.2: L1 Requirements Document

The skill shall instruct the agent to create or update an `L1.md` file in `docs/specs` containing high-level requirements for the ask.

**Acceptance Criteria:**

- The skill file contains an instruction to produce `docs/specs/L1.md`.
- The instruction specifies that L1 requirements are extreme high-level descriptions of what the system shall do.
- Each L1 requirement has a unique identifier (e.g., `L1-1`, `L1-2`).
- The instruction specifies that if `L1.md` already exists, the agent shall update it rather than overwrite it.

### L2-3.3: L2 Requirements Document

The skill shall instruct the agent to create or update an `L2.md` file in `docs/specs` containing detailed requirements that trace to L1 requirements.

**Acceptance Criteria:**

- The skill file contains an instruction to produce `docs/specs/L2.md`.
- Each L2 requirement traces to exactly one L1 requirement via its identifier.
- Each L2 requirement includes acceptance criteria defining done.
- The instruction specifies that if `L2.md` already exists, the agent shall update it rather than overwrite it.

### L2-3.4: Append-Only Requirement Updates

When L1.md and L2.md already exist, the skill shall instruct the agent to make only append-only changes — existing requirements shall not be removed or have their meaning reduced.

**Acceptance Criteria:**

- The skill file contains an instruction that changes to existing requirements are append-only.
- Existing L1 and L2 requirements are never deleted when updating.
- Existing acceptance criteria are never removed when updating.
- New requirements and acceptance criteria may be added freely.
- Existing requirement text may be updated to broaden scope (e.g., "create" becomes "create or update") but never to narrow it.

### L2-3.5: Existing File Detection

The skill shall instruct the agent to check whether `docs/specs/L1.md` and `docs/specs/L2.md` already exist before generating requirements, and behave accordingly.

**Acceptance Criteria:**

- The skill file contains an instruction to read existing L1.md and L2.md if they exist.
- If the files exist, the agent updates them in place rather than creating new files from scratch.
- If the files do not exist, the agent creates them as new files.

## Traces to L1-4: Spec Skill — Team-Based Implementation Process

### L2-4.1: Per-L1 Team Creation

The skill shall instruct the agent to create a team of agents for each L1 requirement.

**Acceptance Criteria:**

- The skill file describes creating a team per L1 requirement.
- Each team consists of at least an Implementor agent and a Quality Assurance agent.

### L2-4.2: Implementor Agent Behavior

The Implementor agent shall systematically implement each L2 requirement for its assigned L1, using test-driven development.

**Acceptance Criteria:**

- The skill file instructs the Implementor to iterate through each L2 under the assigned L1.
- For each L2, the Implementor creates a failing acceptance test for each acceptance criterion before writing production code.
- The Implementor writes the minimal code to make each acceptance test pass.
- The Implementor marks each L2 requirement as complete when all its acceptance tests pass.

### L2-4.3: Quality Assurance Agent Behavior

The Quality Assurance agent shall verify each completed L2 requirement.

**Acceptance Criteria:**

- The skill file instructs the QA agent to review each L2 marked as complete.
- The QA agent verifies the implementation is complete with no TODOs or placeholder code.
- The QA agent verifies all acceptance tests pass.
- The QA agent marks the L2 as verified, or rejects it with feedback so the Implementor can fix it.

### L2-4.4: Iteration Until Verified

The team shall iterate until all L2 requirements under the L1 are verified.

**Acceptance Criteria:**

- The skill file specifies that the Implementor and QA agents loop until every L2 is marked as verified.
- Rejected L2 requirements are reworked by the Implementor and re-submitted for QA verification.
- The process completes only when all L2 requirements for the L1 are in a verified state.
